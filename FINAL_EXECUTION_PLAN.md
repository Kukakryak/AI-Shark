# FINAL EXECUTION PLAN

## 1. Главная мысль

Проект как кодовая база уже почти готов.

Финальный этап сейчас состоит не в переписывании архитектуры, а в правильной постановке итоговых экспериментов для диплома.

Тебе нужно сделать не "ещё одну нейросеть", а:

1. выбрать корректные сценарии;
2. запустить финальные эксперименты;
3. сохранить артефакты;
4. собрать таблицы и графики;
5. встроить выводы в текст диплома.

## 2. Какие сценарии стоит брать в диплом

Ниже разделение на:

- обязательные;
- дополнительные;
- нежелательные.

## 2.1 Обязательные сценарии

### Сценарий 1. `bot_intraday`

Файл:

- `03-02-2018.csv`

Классы:

- `Benign`
- `Bot`

Почему брать:

- это самый адекватный из уже проверенных сценариев;
- в нём test не полностью вырожден;
- на нём хорошо видно сравнение `LSTM vs baseline`.

Зачем он нужен в дипломе:

- это твой основной "показательный" эксперимент.

### Сценарий 2. `02-16-2018` как intraday multi-class

Файл:

- `02-16-2018.csv`

Классы:

- `Benign`
- `DoS attacks-Hulk`
- `DoS attacks-SlowHTTPTest`

Почему брать:

- это более интересный сценарий, чем дни почти с одним классом;
- там есть уже не только бинарная, но почти многоклассовая постановка;
- распределение заметно лучше, чем в `02-15`.

Зачем он нужен:

- показать, что проект способен решать не только `Benign vs one attack`.

### Сценарий 3. `02-14-2018` как intraday brute force

Файл:

- `02-14-2018.csv`

Классы:

- `Benign`
- `FTP-BruteForce`
- `SSH-Bruteforce`

Почему брать:

- у этого дня есть 3 класса;
- на полном датасете это уже интереснее, чем быстрый урезанный прогон.

Важно:

- именно на полном датасете, не на первых `20000` строках.

## 2.2 Дополнительные сценарии

### Сценарий 4. `infiltration_crossday`

Файлы:

- train: `02-28-2018.csv`
- test: `03-01-2018.csv`

Классы:

- `Benign`
- `Infilteration`

Почему можно брать:

- это уже cross-day постановка;
- она сильнее с точки зрения исследования, чем intraday.

Почему это не главный сценарий:

- cross-day в CIC-IDS надо интерпретировать аккуратно;
- лучше брать его как дополнительное подтверждение, а не как основной результат.

### Сценарий 5. `02-20-2018`

Файл:

- `02-20-2018.csv`

Классы:

- `Benign`
- `DDoS attacks-LOIC-HTTP`

Почему можно взять:

- большой объём;
- чистая бинарная постановка;
- хороший технический стресс-тест.

## 2.3 Сценарии, которые лучше не делать основными

### `02-15-2018`

Причина:

- в быстрых прогонах он давал почти моноклассовый test;
- метрики там легко становятся методологически слабыми.

### `02-22-2018` и `02-23-2018`

Причина:

- web/XSS/SQL там слишком редкие;
- сильнейший дисбаланс;
- для диплома это можно использовать разве что как пример сложного случая, но не как главный результат.

### `02-21-2018`

Причина:

- `DDOS attack-LOIC-UDP` слишком редкий;
- интерпретация результатов будет слабой.

## 3. Что надо сделать обязательно перед финальными запусками

## 3.1 Не использовать `--fast-max-rows`

Для финальных результатов диплома нужно запускать без искусственного обрезания строк.

`--fast-max-rows` годится только для:

- отладки;
- smoke-run;
- предварительного сравнения.

## 3.2 Сохранять результаты по отдельным папкам

Для каждого финального сценария лучше иметь отдельный каталог.

Пример:

- `reports_final/bot_intraday_full/`
- `reports_final/bruteforce_intraday_full/`
- `reports_final/slowhttp_hulk_full/`
- `reports_final/infiltration_crossday_full/`

## 3.3 Использовать одинаковое число эпох для LSTM

Для честности лучше взять одинаковую настройку, например:

- `epochs = 10` или `epochs = 15`
- `patience = 5`

Если железо позволяет, можно использовать `epochs = 20`.

## 5. Какой минимальный набор результатов нужен в диплом

Для каждого выбранного сценария нужно сохранить:

1. `classification_report`
2. `confusion_matrix`
3. `precision_macro`
4. `recall_macro`
5. `f1_macro`
6. `training_history.png`
7. краткое описание состава классов

## 6. Что вставить в диплом в виде таблиц

Тебе нужны минимум 3 таблицы.

## Таблица 1. Распределение классов по датасетам

Источник:

- `reports/dataset_profile.csv`

В таблице должны быть:

- имя CSV
- классы
- число объектов
- доля класса

Эта таблица нужна, чтобы показать дисбаланс.

## Таблица 2. Сравнение моделей по сценариям

Столбцы:

- сценарий
- модель
- precision macro
- recall macro
- F1 macro

Это главный сравнительный результат.

## Таблица 3. Параметры LSTM

Столбцы:

- длина последовательности
- число LSTM-нейронов
- dropout
- batch size
- epochs
- patience

Это удобно для описания архитектуры.

## 7. Какие графики нужны

Минимум:

1. `training_history.png` для 1-2 ключевых сценариев
2. график сравнения `macro F1` по моделям
3. confusion matrix для лучшего и худшего сценария LSTM

Если делать по уму:

- отдельный столбчатый график `LSTM vs Logistic Regression vs Random Forest`

## 8. Какой честный вывод стоит делать

Главный итог должен звучать примерно так:

"Разработанная LSTM-модель корректно обучается на признаках сетевого трафика и способна решать задачу выявления сетевых атак. Однако сравнительные эксперименты показали, что на ряде сценариев классические методы машинного обучения, такие как Logistic Regression и Random Forest, обеспечивают более высокое качество классификации. Следовательно, использование нейронной сети оправдано как исследовательский подход, но не гарантирует превосходства над baseline-моделями для табличного представления сетевого трафика."

Это хороший, зрелый и честный вывод.

## 9. Что говорить, если спросят "почему LSTM хуже?"

Ответ:

- признаки уже находятся в табличном агрегированном виде;
- часть временной структуры могла быть потеряна ещё на этапе формирования признаков в CIC-IDS;
- для таких данных деревья и линейные модели могут оказаться эффективнее;
- LSTM всё равно была исследована и показала работоспособность, но baseline-сравнение выявило ограничения нейросетевого подхода.
