# PROJECT GUIDE

## 1. Что это за проект

Этот проект решает задачу обнаружения сетевых атак по данным сетевого трафика.

Идея простая:

1. У нас есть CSV-файлы с признаками сетевых соединений.
2. В каждой строке есть числовые характеристики трафика.
3. Для каждой строки есть метка класса `Label`, например:
   - `Benign` - нормальный трафик
   - `FTP-BruteForce`
   - `SSH-Bruteforce`
   - `Bot`
   - `Infilteration`
   - разные варианты `DoS` и `DDoS`
4. Мы хотим научить модель отличать нормальный трафик от атакующего и различать типы атак.

В проекте в качестве основной модели используется `LSTM`.

## 2. Почему здесь LSTM

LSTM - это разновидность рекуррентронной нейронной сети, которая умеет работать с последовательностями.

Почему она вообще может быть полезна здесь:

- сетевой трафик идёт как поток событий;
- соседние записи могут быть связаны во времени;
- модель может учитывать не одну строку, а окно из нескольких подряд идущих строк.

В проекте LSTM получает не одну строку, а последовательность длиной `sequence_length` и предсказывает класс последней строки в этом окне.

Пример:

- есть строки `1..10`;
- модель получает признаки всех этих 10 строк;
- целевая метка - `Label` десятой строки.

Это позволяет модели учитывать контекст предыдущих событий.

## 5. Подробный разбор `preprocessing.py`

## 5.1 `TrainingConfig`

Это dataclass с параметрами обучения.

Его роль:

- хранить все настройки в одном месте;
- не разбрасывать магические числа по коду;
- упростить передачу параметров между функциями.

Поля:

- `dataset_path` - путь к одному CSV, если запускается обычный режим.
- `train_datasets` - список CSV для обучения.
- `test_datasets` - список CSV для тестирования.
- `output_dir` - куда сохранять артефакты.
- `sequence_length` - длина окна для LSTM.
- `test_fraction` - доля test при хронологическом split.
- `epochs` - число эпох обучения.
- `batch_size` - размер батча.
- `validation_fraction` - доля validation внутри train.
- `dropout` - коэффициент dropout.
- `lstm_units` - число нейронов в LSTM-слое.
- `dense_units` - число нейронов в скрытом Dense-слое.
- `early_stopping_patience` - терпение для EarlyStopping.
- `random_seed` - seed для воспроизводимости.
- `max_rows` - ограничение числа строк для быстрых прогонов.

Что говорить на защите:

"Конфигурация вынесена в отдельную структуру, чтобы обеспечить воспроизводимость эксперимента и управляемость гиперпараметров."

## 5.2 `PreparedData`

Это dataclass с уже подготовленными данными.

Он хранит:

- `y_train`, `y_test`
- `scaler`
- имя столбца класса
- mapping классов
- список признаков
- число строк в train/test

Зачем он нужен:

- после подготовки данных мы не передаём 10 переменных по отдельности;
- всё лежит в одном объекте.

## 5.3 `set_global_seed(seed)`

Назначение:

- зафиксировать генератор случайных чисел.

Почему это важно:

- без этого результаты могут заметно гулять от запуска к запуску;
- для диплома нужна воспроизводимость.

Что сейчас делает:

- `random.seed(seed)`
- `np.random.seed(seed)`

Можно улучшить:

- добавить ещё `tensorflow.random.set_seed(seed)`.

## 5.4 `configure_gpu()`

Назначение:

- проверить, доступен ли GPU;
- если доступен, попытаться включить корректное управление памятью.

Почему это нужно:

- TensorFlow иногда пытается сразу занять всю память GPU;
- `set_memory_growth` позволяет выделять память постепенно.

Что делает функция:

1. Пытается импортировать TensorFlow.
2. Если TensorFlow недоступен, возвращает пустой список.
3. Если доступен, получает список GPU.
4. Для первого GPU включает `memory_growth`.
5. Возвращает список имён устройств.

Что важно понимать:

- на Windows с современными версиями TensorFlow GPU часто не используется нативно;
- поэтому в логах часто будет `CPU only`.

## 5.5 `detect_label_column(dataframe)`

Назначение:

- определить, какой столбец является целевой меткой.

Логика:

1. Если есть столбец `Label`, вернуть его.
2. Иначе найти первый строковый столбец и считать его классом.

Зачем это полезно:

- код становится менее жёстко привязан к одному формату данных.

Но practically:

- для CIC-IDS это почти всегда именно `Label`.

## 5.6 `load_dataset(dataset_path, max_rows=None)`

Назначение:

- загрузить один CSV.

Что делает:

1. Читает файл через `pandas.read_csv`.
2. Если есть `Timestamp`, удаляет его.

Почему удаляется `Timestamp`:

- модель дальше работает с числовыми признаками;
- timestamp в текущей реализации не преобразуется в полезный числовой вид;
- оставлять строковую дату нельзя.

Замечание:

Поле времени само по себе может быть полезно, но тогда его нужно отдельно преобразовывать, а не просто подавать как строку.

## 5.7 `load_datasets(dataset_paths, max_rows=None)`

Назначение:

- загрузить несколько CSV и объединить их в один DataFrame.

Что делает:

1. По очереди читает файлы.
2. В каждый добавляет столбец `SourceFile`.
3. Склеивает всё через `pd.concat`.

Зачем нужен `SourceFile`:

- потом можно отделить строки из train-файлов от строк из test-файлов;
- это важно для cross-day экспериментов.

Особенность `max_rows`:

- если задан лимит, строки расходуются по файлам последовательно.
- То есть это режим для быстрых прогонов, а не для строгой научной оценки.

## 5.8 `clean_dataframe(dataframe, label_column)`

Это очень важная функция.

Она очищает данные перед обучением.

Что делает:

1. Копирует DataFrame.
2. Приводит `Label` к строке и убирает пробелы.
3. Удаляет строки, где метка равна `"Label"`.
4. Заменяет `inf` и `-inf` на `NaN`.
5. Удаляет строки с `NaN`.
6. Проверяет, что данные не пусты.
7. Проверяет, что в данных хотя бы 2 класса.

Почему это важно:

- в CIC-IDS встречаются мусорные строки и артефакты;
- нейросеть нельзя обучать на `NaN`;
- one-class training не имеет смысла для многоклассовой классификации.

Что говорить на защите:

"Очищение данных удаляет некорректные значения и предотвращает обучение на повреждённых записях."

## 5.9 `encode_labels(dataframe, label_column)`

Назначение:

- перевести строковые метки классов в целые числа.

Пример:

- `Benign -> 0`
- `Bot -> 1`
- `DoS -> 2`

Почему это нужно:

- нейросеть работает не со строками, а с числовыми метками.

Что возвращает:

- новый DataFrame;
- словарь `label_mapping`.

Почему mapping важен:

- потом нужно понимать, какой номер соответствует какому классу;
- без этого невозможно интерпретировать прогнозы.

## 5.10 `chronological_split(dataframe, test_fraction)`

Назначение:

- делить данные на train/test без случайного перемешивания.

Почему это лучше, чем random split, в этом проекте:

- у нас последовательности;
- LSTM использует окна соседних строк;
- если перемешать, получится утечка данных между train и test.

Что делает:

1. Считает индекс разделения.
2. Берёт первые строки в train.
3. Последние строки в test.

Что говорить на защите:

"Разделение выполнено по времени, а не случайно, чтобы избежать утечки при использовании последовательностной модели."

## 5.11 `split_by_config(dataframe, config)`

Назначение:

- выбрать режим split.

Есть 2 режима:

1. Если заданы `train_datasets` и `test_datasets`, разделение идёт по файлам.
2. Иначе используется хронологический split внутри одного объединённого набора.

Почему это важно:

- для диплома тебе нужны оба варианта:
  - быстрый intraday split;
  - более строгий cross-day split.

## 5.12 `validate_class_coverage(train_df, test_df, label_column)`

Одна из ключевых защитных функций.

Что она проверяет:

1. В test должно быть хотя бы 2 класса.
2. Все классы из test должны присутствовать в train.

Зачем это нужно:

- если в test только один класс, метрики будут обманчиво красивыми;
- если в test есть класс, которого вообще не было в train, многоклассовая оценка становится некорректной.

Почему это особенно важно для CIC-IDS:

- отдельные дни часто содержат очень ограниченный набор атак;
- если выбрать плохую пару train/test, эксперимент будет методологически слабым.

## 5.13 `scale_features(train_df, test_df, label_column)`

Назначение:

- нормализовать признаки в диапазон `[0, 1]`.

Как работает:

1. Выбираются все признаки, кроме `Label` и `SourceFile`.
2. Создаётся `MinMaxScaler`.
3. `scaler.fit_transform` применяется только к train.
4. `scaler.transform` применяется к test.

Почему это правильно:

- scaler нельзя обучать на test, иначе возникает утечка данных.

Что это значит простыми словами:

Модель должна видеть test как новые данные, а не как часть того, на чём уже подстраивалась нормализация.

## 5.14 `build_sequences(dataframe, label_column, sequence_length)`

Это центральная функция для LSTM.

Она превращает табличные данные в последовательности.

Что делает:

1. Убирает столбец класса из признаков.
2. Формирует окно длиной `sequence_length`.
3. Меткой окна делает класс последней строки окна.

Пример:

Если длина окна `10`, то:

- `x[0]` = строки `0..9`
- `y[0]` = класс строки `9`

Результат:

- `x_data` имеет форму `(samples, timesteps, features)`
- `y_data` имеет форму `(samples,)`

Почему форма такая:

- `samples` - сколько окон получилось;
- `timesteps` - длина окна;
- `features` - число признаков в одной строке.

## 5.15 `prepare_data(config)`

Это большая orchestration-функция подготовки данных.

Она вызывает почти все предыдущие шаги.

Есть 2 ветки работы:

### Ветка 1. Явно заданы train/test файлы

1. Загружается train-группа файлов.
2. Загружается test-группа файлов.
3. Находится столбец метки.
4. Очищаются train и test.
5. Формируется общий `label_mapping`.
6. Метки кодируются одинаково и для train, и для test.

### Ветка 2. Обычный режим

1. Загружается один файл или набор train-файлов.
2. Выполняется очистка.
3. Кодируются метки.
4. Выполняется split.

После этого:

1. Проверяется покрытие классов.
2. Масштабируются признаки.
3. Строятся последовательности.
4. Формируется `PreparedData`.

Это одна из главных функций проекта.

## 5.16 `build_model(input_shape, num_classes, config)`

Назначение:

- создать архитектуру нейронной сети.

Структура сети:

1. `Input`
2. `LSTM(config.lstm_units)`
3. `Dropout(config.dropout)`
4. `Dense(config.dense_units, activation="relu")`
5. `Dense(num_classes, activation="softmax")`

Что это значит:

- `LSTM` извлекает временные зависимости из последовательности;
- `Dropout` борется с переобучением;
- `Dense + ReLU` даёт нелинейное преобразование;
- `Softmax` выдаёт вероятности по классам.

Почему `softmax`:

- потому что это многоклассовая классификация;
- сумма вероятностей по классам равна 1.

Почему `sparse_categorical_crossentropy`:

- целевые метки хранятся как числа, а не как one-hot векторы;
- это проще и удобнее.

## 5.17 `compute_class_weights(y_train)`

Назначение:

- компенсировать дисбаланс классов.

Проблема:

- в CIC-IDS классы распределены очень неравномерно;
- нормального трафика обычно намного больше;
- некоторые атаки очень редкие.

Решение:

- вычислить веса классов;
- редким классам дать больший вес в функции потерь.

Что это даёт:

- модель меньше игнорирует редкие атаки.

## 5.18 `train_model(prepared, config)`

Назначение:

- обучить нейросеть.

Что делает:

1. Создаёт `EarlyStopping`.
2. Строит модель.
3. Запускает `model.fit(...)`.

Параметры обучения:

- `batch_size`
- `epochs`
- `validation_split`
- `class_weight`

Что такое `EarlyStopping`:

- если ошибка на validation перестаёт улучшаться, обучение останавливается;
- восстанавливаются лучшие веса.

Почему это хорошо:

- меньше риск переобучения;
- не тратится лишнее время на бесполезные эпохи.

## 5.19 `evaluate_model(model, prepared)`

Назначение:

- посчитать качество модели на test.

Что делает:

1. Получает вероятности классов через `model.predict`.
2. Берёт `argmax`, то есть класс с максимальной вероятностью.
3. Сравнивает предсказания с истинными метками.
4. Считает:
   - `precision_macro`
   - `recall_macro`
   - `f1_macro`
   - `classification_report`
   - `confusion_matrix`

Что значат метрики:

- `Precision` - насколько точны предсказания данного класса;
- `Recall` - сколько реальных объектов класса модель смогла найти;
- `F1` - баланс precision и recall;
- `Macro` - среднее по классам без учёта их размера.

Почему именно macro-метрики важны:

- при сильном дисбалансе accuracy часто врёт;
- macro-F1 показывает качество по классам более честно.

## 5.20 `plot_history(history, output_path)`

Назначение:

- сохранить графики обучения.

Строятся 2 графика:

1. Accuracy
2. Loss

Зачем они нужны:

- показать, как шло обучение;
- увидеть переобучение или недообучение.

Как читать графики:

- если train accuracy растёт, а val accuracy падает, это переобучение;
- если и loss, и accuracy почти не меняются, модель недообучается или признаки плохие.

## 6.1 `ExperimentScenario`

Это структура одного сценария эксперимента.

Поля:

- `name` - имя сценария;
- `description` - текстовое описание;
- `train_datasets` - train-файлы;
- `test_datasets` - test-файлы;
- `dataset` - резервный путь;
- `max_rows` - ограничение размера.

Идея:

сценарий - это готовый воспроизводимый эксперимент.

## 6.2 `SCENARIOS`

Это заранее заданный список сценариев.

Сейчас там есть:

- `bruteforce_intraday`
- `dos_intraday`
- `slowhttp_vs_hulk_intraday`
- `web_attacks_crossday`
- `infiltration_crossday`
- `bot_intraday`

Что это означает:

- `intraday` - обучение и тест внутри одного дня, через хронологический split;
- `crossday` - обучение на одном дне, тестирование на другом.

Это удобно для диплома, потому что можно сравнивать режимы эксперимента.

## 6.3 `profile_datasets(output_path)`

Назначение:

- построить профиль датасетов по классам.

Что делает:

1. Для каждого CSV читает только столбец `Label`.
2. Удаляет мусорные строки `Label`.
3. Считает число объектов каждого класса.
4. Сохраняет таблицу в `dataset_profile.csv`.

Зачем это нужно:

- понять распределение классов;
- понять, какие дни годятся для train/test;
- показать в дипломе дисбаланс данных.

Это очень важная часть аналитики.

## 6.4 `build_config(scenario, output_dir, epochs, batch_size)`

Назначение:

- преобразовать сценарий эксперимента в `TrainingConfig`.

То есть это мост между `experiment.py` и `preprocessing.py`.

## 6.5 `run_scenarios(selected_names, epochs, batch_size, fast_max_rows)`

Назначение:

- запустить серию сценариев.

Что делает:

1. Проходит по сценариям.
2. Если задан фильтр, запускает только выбранные.
3. Строит конфиг.
4. При необходимости подменяет `max_rows` быстрым лимитом.
5. Запускает `run_training`.
6. Читает `metadata.json`.
7. Собирает итоговую таблицу.
8. Если сценарий падает, записывает ошибку в summary.

Почему это удобно:

- можно запустить сразу несколько экспериментов;
- все результаты попадают в единый отчёт.

## 6.6 `parse_args()` и `main()` в `experiment.py`

Поддерживаются аргументы:

## 11. Как объяснять проект на защите

Ниже короткая логика, которую можно использовать как основу речи.

### 11.1 Формулировка задачи

"Цель проекта - разработать систему интеллектуального анализа сетевого трафика, которая по признакам сетевых соединений определяет, является ли трафик нормальным или относится к одному из известных типов атак."

### 11.2 Почему выбран CIC-IDS 2018

"Этот датасет содержит реальные сетевые признаки и множество сценариев атак, поэтому подходит для обучения и оценки систем обнаружения вторжений."

### 11.3 Почему выбран LSTM

"В отличие от обычной табличной классификации, LSTM учитывает последовательность записей и может извлекать временной контекст из потока сетевого трафика."

### 11.4 Почему нельзя было делать random split

"Потому что модель работает с последовательностями. Случайное перемешивание приводит к утечке данных между train и test, так как соседние окна сильно пересекаются."

### 11.5 Почему используется normalisation

"Признаки имеют разные масштабы. Без нормализации крупные по значению признаки начинают доминировать, что ухудшает обучение сети."

### 11.6 Почему используются class weights

"Датасет несбалансирован, и без дополнительного взвешивания модель была бы склонна игнорировать редкие типы атак."

### 11.7 Почему важен macro-F1

"Accuracy на несбалансированных данных может выглядеть очень высокой даже при плохом распознавании редких атак. Macro-F1 лучше отражает качество по всем классам."

## 12. Какие у проекта сейчас сильные стороны

- есть сохранение артефактов;
- есть базовая воспроизводимость;
- есть профилирование датасетов;
- есть экспериментальные сценарии;
- есть unit-тесты на ключевые шаги препроцессинга.

## 13. Какие у проекта сейчас слабые стороны

Их надо честно понимать.

### 13.1 Не все дни можно напрямую сравнивать

Причина:

- в разные дни CIC-IDS содержатся разные attack families.

Следствие:

- не всякая пара `train/test` методологически корректна.

### 13.2 LSTM не обязательно лучший baseline

Для табличных признаков часто очень сильны:

- Logistic Regression
- Random Forest
- XGBoost
- LightGBM

Для диплома это скорее плюс:

можно добавить сравнение и показать, почему LSTM лучше или хуже.

### 13.3 `max_rows` - это режим отладки, а не научной оценки

Если использовать только первые строки файла, можно получить плохое распределение классов.

### 13.4 Пока нет отдельного inference-script

То есть модель умеет обучаться и оцениваться, но отдельной прикладной утилиты "предсказать на новом CSV" пока нет.
